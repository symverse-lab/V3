# 1. SCT

## 1.1 SCT (SymVerse Contract Template)

SCT contains commonly used smart-contract functions so that contract-based services can be implemented simply and efficiently.

### Features

- SCT contains commonly used smart-contract functions and is designed to be simple and easy to use.
- Improved development environment
  - Does not rely on a smart-contract-specific development language such as Solidity.
  - Can be developed through simple API calls, reducing development time.
- Reduced maintenance costs
  - Minimizes security issues caused by custom contract development.

---

# 2. SCT Types

| SCT Type | Description | Representative Uses |
|---|---|---|
| `SCT20`, `SCT21` | ERC20-like token contracts | Tradeable tokens, utility tokens, digital currency, ICO, IEO |
| `SCT30` | ERC721-like ownership representation | Digital or physical assets, housing, illustrations, art, NFTs, game items, loans, repayment burdens, liabilities |
| `SCT40` | Coupon/process contract | Coupon agreements, individual coupon management, expiration conditions, discount coupons, accommodation coupons |
| `SCT50`, `SCT51` | Voting-related contracts | Vote contract creation, poll creation, voting, poll management |

# 3. SCT Terminology

| Term | Meaning |
|---|---|
| `Token` | Coin generated by `SCT20` |
| `Creator` | Contract creator of an SCT contract; master owner |
| `Owner` | Owner account of the SCT contract |
| `Spender` / `Delegator` | Account authorized to transfer part of the Owner’s tokens |
| `User` / `Holder` | Any account that has user rights |

# 4. Delegation of Authority

## 4.1 Creator

The Creator can appoint and revoke Owner authority for one account.

| Action | Function |
|---|---|
| Appoint | `SCT*_APPROVE` |
| Dismiss | `SCT*_DECREASE_APPROVE` |

When creating the initial contract, the Creator must designate:

- itself as Owner, or
- another account as Owner.

---

## 4.2 Owner

The Owner has some rights delegated from the Creator.

---

## 4.3 Spender (Delegator)

Any account holding tokens, including User, Owner, or Creator, can assign another account as a Spender that receives remittance authority for a portion of those tokens.

A Spender may send tokens to other accounts up to the delegated amount.

### Example

Assume:

```text
Owner A
User B
User C
```

and:

```text
A owns 100 tokens.
```

Then:

```text
A appoints B as Spender
and delegates 50-token remittance authority.
```

B may then send:

```text
10 of the 50 delegated tokens
from A to C.
```

using:

```text
SCT*_TRANSFER_FROM
```

### Delegation Restrictions

- B cannot appoint C as another Spender.
- A Spender cannot delegate remittance authority to another account.
- When B sends delegated tokens to another account, the transaction uses B’s private key, but the actual token transfer is performed from A to C.

### Why Spender Delegation Is Used

Delegating a Spender account is useful for two main reasons.

1. It allows a token holder to manage tokens through another operational account.
2. It reduces private-key risk for application servers.  
   If an Owner designates a Spender and operates an application server with that Spender account, the Owner can replace the Spender if the application server’s private key is lost or leaked.

# 6. SCT Function and Gas

SYM Coin is required to create and use SCT contracts.  
Gas Fee is the amount of SYM Coin required to call an SCT function.

## 6.1 Total Gas

```text
Total Gas =
  Transaction Gas
  + Create Gas
  + Data Gas
  + SCT Gas
```

---

## 6.2 Transaction Gas

```text
if block >= 4,745,000:
  Transaction Gas = 49,000
else:
  Transaction Gas = 21,000
```

---

## 6.3 Create Gas

```text
if To != nil:
  Create Gas = 800,000
```

---

## 6.4 Data Gas

```text
if block >= 4,745,000:
  Data Gas = NonZeroGas(680) × NonZeroSize
           + ZeroGas(40) × ZeroSize
else:
  Data Gas = NonZeroGas(68) × NonZeroSize
           + ZeroGas(4) × ZeroSize
```
